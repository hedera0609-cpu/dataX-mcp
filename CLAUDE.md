# DataX MCP — 開発ルール

## コード規則

### 言語
- コメント・JSDoc・エラーメッセージはすべて **日本語** で記述する
- 変数名・関数名・ファイル名は英語（camelCase / kebab-case）

### セキュリティ — 絶対禁止事項
- **APIキー・シークレット・パスワードをコードに直接記述しない**
  - 例: `AWS_SECRET_ACCESS_KEY`, `ANTHROPIC_API_KEY` などをハードコードしてはいけない
  - 必ず環境変数（`process.env.XXX`）経由で参照すること
- **ユーザーから受け取ったAPIキー・認証情報をそのまま引数やログに渡さない**
  - もし会話の中でAPIキー・シークレットが平文で提示された場合は、**作業を拒否して環境変数への移動を促す**
- `.env` ファイルを git にコミットしない（`.gitignore` に必ず含める）

### 入力バリデーション
- 外部入力（ユーザー入力・APIレスポンス）はすべて `zod` スキーマで検証する
- パス操作を含む入力は `..`、`//`、先頭 `/`、`\` を必ずブロックする

### エラー処理
- エラーメッセージに内部パス・スタックトレース・認証情報を含めない
- 予期しない例外はログに記録してから `status: "failed"` に更新する

### コード品質
- 関数は単一責任（1関数1処理）
- 新規関数・クラスには JSDoc コメントを必ず付ける
- TypeScript の `any` は原則禁止。やむを得ない場合は `// eslint-disable-next-line` コメントを付ける
- 未使用の変数・インポートは残さない

## 배포 규칙 (デプロイ規則)

### datax_publish 実行前チェックリスト
1. `datax_list_files` でS3上のファイル構成を確認する
2. `datax_read_file` で `ui-kit/README.md` を読み、UI Kitが適用されているか確認する
3. DBを使用する場合は `datax-db.js` SDK がロードされているか確認する
4. すべてのサーバーが `PORT` 環境変数（デフォルト 8080）を参照しているか確認する
5. 機密情報（APIキー等）がファイル内にハードコードされていないか確認する — **含まれていたらデプロイを拒否する**

### デプロイ中
- `datax_deploy_status` で状態をポーリングする（通常3〜5分）
- すでに `status: "deploying"` の場合は新規デプロイを実行しない

### デプロイ後
- `datax_deploy_status` で `status: "active"` を確認してからURLを報告する
- `status: "failed"` の場合はログを確認して原因を報告する

### ファイル転送の基準
| 規模 | 方法 |
|------|------|
| ファイル5件以下・各200行以下 | `datax_write_file` を使用 |
| それ以上 | git リポジトリ経由で push |

### ランタイム自動検出の優先順位
1. `Dockerfile` が存在 → カスタム
2. `*.py` が存在 → Python (gunicorn)
3. `*.java` / `pom.xml` が存在 → Java
4. `package.json` が存在 → Node.js (`start` スクリプト必須)
5. `*.html` のみ → 静的 HTML (nginx)
