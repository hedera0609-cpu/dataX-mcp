01   시스템 개요

DataX MCP란

비기술자(개발 지식이 없는 사용자)가 Claude AI를 통해 안전하게 웹 애플리케이션을 개발·배포할 수 있도록 설계된 플랫폼입니다.

핵심 설계 목표는 사용자가 보안 규칙을 몰라도, 시스템이 자동으로 위반을 방지하는 것입니다.

03   현재 리스크 평가 (솔직한 평가)
Critical

파일 내용 무검증 배포

비기술자가 Claude에게 "API키를 출력하는 앱을 만들어줘"라고 하면, datax_write_file이 내용을 검사하지 않고 S3에 저장 → 인터넷에 배포됩니다.

High

감사 로그 부재

누가 언제 무엇을 배포했는지 추적이 불가능합니다. 사고 발생 시 원인 규명이 어렵고 컴플라이언스 요건을 충족하지 못합니다.

High

관리자 기능 부재

보안 관리자가 전체 배포 현황을 볼 수 없고, 문제 앱을 강제 삭제할 수단이 없습니다. 현재 datax_list는 본인 앱만 반환합니다.

Medium

컨테이너 네트워크 egress 무제한

배포된 ECS 컨테이너가 외부 서버로 데이터를 전송하거나 내부 네트워크를 스캔하는 코드를 포함해도 막을 수 없습니다.

Medium

AWS 키 평문 저장

~/.claude/settings.json에 AWS 인증정보가 평문으로 저장됩니다. MCP 설정 방식의 구조적 한계이며, PC 탈취 시 키 노출 위험이 있습니다.

Low

ALB HTTP 통신

현재 검증 단계로 HTTPS 미적용입니다. 도메인 연결 시 ACM 인증서 + HTTPS 전환이 필요합니다.

04   개선 로드맵
P1

파일 업로드 시 시크릿 스캔 진행중

datax_write_file 단계에서 파일 내용에 AWS 키, API 키, 패스워드 패턴이 탐지되면 저장 자체를 차단합니다. CLAUDE.md 소프트 룰을 하드 룰로 격상.

// 탐지 패턴 예시 AKIA[0-9A-Z]{16} // AWS Access Key -----BEGIN.*PRIVATE KEY----- // 개인키 password\s*=\s*['"][^'"]{8,} // 패스워드 하드코딩 api_key\s*=\s*['"][^'"]{8,} // API 키
P2

관리자 도구 추가

환경변수 DATAX_ADMIN_NICKNAME을 지정한 관리자 계정에 한해 전체 앱 조회 및 강제 삭제 권한 부여.

datax_admin_list // 전체 사용자 앱 조회 datax_admin_delete // 강제 삭제
P3

감사 로그 (Audit Log)

모든 배포·삭제·파일 업로드 작업을 CloudWatch Logs 또는 DynamoDB 별도 테이블에 기록. 사용자, 타임스탬프, 작업 내용, 결과를 포함.

P4

네트워크 egress 제한

ECS Security Group 아웃바운드 규칙을 강화하여 컨테이너가 허용된 엔드포인트 외부로 통신하는 것을 차단.

05   예상 질문 & 답변

Q. 비기술자가 민감정보를 포함한 코드를 배포하면?

현재: P1 구현 전에는 막을 수 없음.
P1 완료 후: datax_write_file 단계에서 패턴 탐지 시 저장 자체가 차단되고, 사용자에게 환경변수 사용을 안내.

Q. AWS 키가 settings.json에 평문인 이유는?

Claude Code MCP 설정 방식의 구조적 한계. 현재는 파일 시스템 권한으로만 보호. 향후 AWS IAM Role 기반 인증 또는 SSO로 전환 예정.

Q. 사용자 A가 사용자 B의 앱을 삭제할 수 있나?

불가능. DynamoDB PK가 USER#{nickname}으로 구성되어 쿼리 자체가 본인 데이터만 반환. 닉네임은 서버 환경변수에서 고정 주입되어 변조 불가.

Q. 배포된 컨테이너에서 내부 네트워크 스캔이 가능한가?

현재는 egress 제한이 없어 이론적으로 가능. P4 로드맵에서 Security Group 강화로 대응 예정.

Q. 감사 추적(Audit Trail)은?

현재 DynamoDB에 배포 상태와 타임스탬프만 보관. 체계적인 감사 로그는 P3 로드맵. 즉각 조치가 필요하다면 DynamoDB Streams + CloudWatch 연동으로 단기 구현 가능.

06   리뷰 시작 시 오프닝 멘트
"이 시스템은 개발 지식이 없는 사용자가 Claude AI를 통해 앱을 배포할 수 있도록 설계되었습니다.

핵심 보안 원칙은 세 가지입니다. 첫째, 사용자 인증정보는 서버 환경변수에 고정되어 AI를 포함한 누구도 변조할 수 없습니다. 둘째, 모든 입력값은 스키마 검증과 경로 트래버설 차단으로 2중 방어합니다. 셋째, AWS 리소스는 VPC 내부에 격리되고 ALB를 통해서만 외부에 노출됩니다.

다만 파일 내용 스캔, 감사 로그, 관리자 기능은 현재 미흡하며, 오늘 리뷰에서 우선순위를 함께 정하고 싶습니다."
